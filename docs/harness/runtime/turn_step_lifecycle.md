# Turn Step Lifecycle

状态：Current

## 目的

这篇文档解释一个 turn 内的单次 step 大体会经历哪些时段。读者如果想先理解“模型思考一次、可能调用工具、再把工具结果交回模型”这件事的运行节奏，而不是立刻钻进代码细节，应该读这一篇。

## 设计理念

一个 step 可以理解为 runtime 的最小决策循环。它不是完整 turn，也不是某个工具的执行细节，而是把一次模型请求、一次结果判定、可能发生的工具执行和下一轮上下文衔接串起来。这样的拆法能让读者先看清主干：模型先决定要不要回答，runtime 再决定要不要执行工具，工具结果最后再回到模型上下文里。

## 链路位置

这一层位于 [Turn Execution](turn_execution.md) 的核心循环内部。它会连接 [LLM Integration](../llm/streaming_adapter.md)、[Tool Execution](tools/tool_execution.md) 和 [上下文裁剪与压缩](context/context_pruning_compaction.md)，但不展开这些专题的内部实现。本文只说明一个 step 如何在这些子系统之间传递控制权。

## 总览

一个 step 可以按七个时段理解：进入时先确认本轮还能不能继续；随后向 LLM 发起请求；拿到结果后判断是直接回答、执行工具、恢复上下文还是退出；如果需要工具，先提交工具调用意图，再执行工具；工具结果写回后，下一次 step 才能继续基于这些结果推理。

不是每个 step 都会进入工具相关时段。模型如果已经给出最终文本，turn 会在当前 step 结束；只有模型选择调用工具时，runtime 才会进入工具提交、工具执行和结果回写。

## Step 进入时段

### 目的

进入时段决定当前 step 是否还有继续执行的必要，以及是否应该继续允许模型拉长工具链。

### 设计理念

runtime 在每次开始前先处理全局边界，是为了避免已经取消的 turn 继续消耗资源，也避免一个 turn 无限延伸。越接近执行上限，runtime 越倾向于要求模型收束成最终回答，而不是继续请求更多工具。

### 发生事件

这个时段会先检查 turn 是否已经被取消。取消意味着用户或外部调度已经不再需要本轮继续执行，因此 step 不会再发起新的模型请求。

如果 turn 仍可继续，runtime 会判断当前是否已经接近单轮执行上限。普通 step 会把可用工具继续交给模型；最后的收束 step 会更强调输出最终文本，减少继续调用工具的机会。

## LLM 请求时段

### 目的

LLM 请求时段负责把当前 step 所需的信息交给模型，并把模型的流式响应收束成 runtime 能理解的一次结果。

### 设计理念

step 本身不关心不同供应商的通信细节。它只关心三件事：这次请求有没有成功，模型有没有产生可见文本，模型有没有请求工具。供应商协议、流式事件、thinking、trace 和取消句柄都由 LLM 集成层处理，step 只接收归一化后的结果。

### 发生事件

runtime 会带着当前上下文、系统提示、可用工具说明和工具选择策略向 LLM 发起请求。请求过程中，流式文本可以被转发给最终回复展示层，thinking 和工具请求等事件也会被收集成 step 级摘要。

如果请求遇到可重试的临时问题，LLM 集成层会在本 step 内尝试恢复。如果遇到不可恢复的问题，或供应商明确表示上下文过大，控制权会回到 step，由 step 决定是进入错误出口还是尝试上下文恢复。

LLM 供应商适配的细节见 [Streaming Adapter](../llm/streaming_adapter.md)。

## LLM 结果判定时段

### 目的

结果判定时段决定当前 step 接下来走哪条路：直接结束 turn、进入工具执行、尝试上下文恢复，或带着错误退出。

### 设计理念

runtime 把模型结果分成少数几类，是为了让 turn 的控制流稳定可解释。最终文本意味着本轮可以收束；工具调用意味着模型还需要外部事实或副作用；上下文过载意味着历史需要压缩后再试；普通错误则应该尽快转成可持久化、可展示的失败结果。

### 发生事件

如果模型返回的是最终文本，runtime 会把这段文本作为本 turn 的回答，并结束当前 turn。

如果模型返回的是工具调用，runtime 不会立刻再次请求模型，而是先进入工具相关时段。因为模型此时表达的是“我需要这个工具结果之后才能继续”。

如果模型响应表示上下文过大，runtime 会优先尝试压缩旧历史，并用压缩后的上下文重新进入本 step。只有恢复失败时，才会把它当作错误处理。

如果是普通模型错误，runtime 会生成错误结果，让 turn 以可解释的方式结束，而不是把半截状态留在历史里。

Turn 内上下文恢复的细节见 [上下文裁剪与压缩实现细节](context/context_pruning_compaction.md)。

## Assistant Tool Call 提交时段

### 目的

这个时段把模型“想调用什么工具”的意图先写入本轮历史。它让后续工具结果有明确的对应对象，也让失败和取消场景仍然能解释清楚。

### 设计理念

工具调用先入历史，工具结果后入历史，这是一种闭合结构。它保证 runtime 不会只留下一个结果，却看不到模型为什么请求它；也不会只留下一个请求，却没有后续观察结果。先提交意图，再执行工具，是为了让对话历史始终能被下一次模型请求理解。

### 发生事件

runtime 会记录模型这次提出的工具调用意图，包括工具名称、参数轮廓和模型当时的公开说明。然后它会把这次意图写成本轮 assistant 消息的一部分。

从这个时刻开始，历史中已经存在一个待闭合的工具调用。后面无论工具成功、失败、未知还是被取消，runtime 都需要写回对应的 tool result，让历史保持完整。

## Tool 执行时段

### 目的

Tool 执行时段负责把模型提出的工具调用变成实际动作，并把动作结果统一整理成模型能继续阅读的观察结果。

### 设计理念

step 只负责运行时边界，不负责每个工具的业务细节。它会确认工具是否存在、展示工具运行状态、感知取消、收集成功或失败结果。具体工具怎样读取文件、执行命令、调用浏览器或处理后台任务，则交给工具专题维护。

### 发生事件

runtime 会按模型给出的顺序处理工具调用。每个工具开始前都会再次检查 cancel，因为工具执行往往会产生副作用或消耗较长时间。

如果工具不存在，runtime 会把它当作一次可解释的工具失败，并把失败信息返回给模型。这样模型可以在下一次 step 中修正工具选择，而不是让 turn 直接断裂。

如果工具存在，runtime 会标记它正在运行，执行对应处理逻辑，并在结束时记录成功或失败。成功结果会成为模型可读的观察内容；失败结果也会被写成观察内容，让模型知道这条路径没有完成。

工具执行的完整生命周期见 [Tool Execution](tools/tool_execution.md)。

## Tool Result 回写时段

### 目的

Tool result 回写时段把工具执行后的观察结果放回本轮历史，让下一次 step 能基于真实结果继续推理。

### 设计理念

模型调用工具不是 turn 的终点，而是中间取证。runtime 必须把取证结果写回上下文，再把控制权交给下一次 step。这样模型才能根据工具结果决定是继续调用工具、修正策略，还是给用户最终回答。

### 发生事件

runtime 会把本 step 中所有工具的观察结果整理成一条工具结果消息，并追加到当前 turn 的历史中。这个消息会和前面的工具调用意图对应起来，形成“模型请求工具，runtime 返回观察”的闭合结构。

写回完成后，当前 step 通常不会直接发送最终回复。下一次 step 会把这些工具结果作为上下文输入，再由模型判断后续动作。

## Step 异常出口时段

### 目的

异常出口时段保证 turn 在取消、模型错误、工具错误、未知工具或执行上限到达时，仍然能留下完整、可解释的历史状态。

### 设计理念

异常处理的核心不是掩盖失败，而是避免留下半截对话。尤其是模型已经提出工具调用后，如果没有对应的工具结果，下一次模型请求可能无法理解历史。runtime 因此会尽量补齐工具结果、生成清晰错误说明，并把控制权交还给 turn 级持久化和发送流程。

### 发生事件

如果 turn 被取消，runtime 会停止继续请求模型或执行新的工具。若已经存在未闭合的工具调用，它会补一个 synthetic cancel 结果，说明这次工具没有机会完成。

如果工具未知或工具执行失败，runtime 会把失败作为 tool result 写回，而不是静默丢弃。这样模型在下一次 step 中仍能看到失败原因。

如果模型请求失败且无法恢复，runtime 会把它转成一次可展示的错误结果，让 turn 能正常结束。

如果 step 链条已经到达本轮执行上限，runtime 会让本 turn 收束成一条提示性回复，而不是继续无限等待更多工具和模型轮次。

这些出口最终都会回到 [Turn Execution](turn_execution.md) 描述的持久化与 final emit 阶段。

我要引入两个新的机制：
1. 对 tool_result 的裁剪。目前 agent loop 中，只要大模型返回 tool_use 类型的文本，那么当前 step 就一定会执行 tool。而 tool 的 result 的大小是不确定的，那么就正好在这个时间段做好管理准备。
触发时间段：tool result 回来后、下一次模型请求前。
第一步：检查 tool_result。制作一个标准，tool result 不能超过 10000 token。如果超过了，就要进行一些特殊处理。具体如下：
```text
把预算按 byte 一分为二：left = budget / 2，right = budget - left。
保留开头最多 left bytes，保留结尾最多 right bytes。
中间删除，并插入 marker：byte 模式：…N chars truncated…
token 模式：…N tokens truncated…

UTF-8 会按字符边界调整，不会切坏 emoji/中文字符。
它不是按行截，也不是按真实 tokenizer 截；token 是 ceil(bytes / 4) 的近似。
```

第二步：在给供应商发送请求前，检查目前 system 和 messages 和 tool schema 加起来的 token 数量，如果超过 90 % 进入压缩处理。
