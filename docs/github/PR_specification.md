比如：
- 目的是什么
- 对项目影响是什么
- 哪些文件发生改变

2. 一个 PR 尽可能只做一件事。
- 单一职责： 避免将功能开发、Bug 修复和代码重构混合在一个 PR 中。
- 命名格式：不同的 PR 要写好前缀，具体参考下面说明

## 1. 标准结构
一条约定式提交信息由 **Header**、**Body** 和 **Footer** 组成：

```text
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

### 核心字段说明：
* **Type (必选)**：说明本次提交的意图（见下文分类）。
* **Scope (可选)**：说明影响的范围（如：`auth`, `parser`, `api`）。
* **Description (必选)**：简洁的描述，通常使用祈使句（如 "add" 而不是 "added"）。
* **Body (可选)**：详细的修改背景、动机、与之前逻辑的区别。
* **Footer (可选)**：用于关联 Issue 或标注 **BREAKING CHANGE**。

---

## 2. 常见的类型 (Types)
根据约定，最常用的 `type` 包括：

| 类型 | 说明 | 对应版本变更 |
| :--- | :--- | :--- |
| **feat** | 新增功能 (Feature) | `Minor` |
| **fix** | 修复 Bug | `Patch` |
| **docs** | 仅文档修改 | 无 |
| **style** | 不影响代码含义的修改（空格、格式化、分号等） | 无 |
| **refactor** | 代码重构（既不是修复 Bug 也不是新增功能） | 无 |
| **perf** | 提高性能的修改 | `Patch` |
| **test** | 添加或修改测试用例 | 无 |
| **chore** | 构建过程或辅助工具的变动（如更新依赖） | 无 |
| **ci** | 持续集成配置文件或脚本的变动 | 无 |

---

### ✅ 好的提交信息 (Conventional Commits)
* `feat(api): 增加 Amazon 订单自动抓取接口`
* `fix(proxy): 修复 IP 切换时的连接泄漏问题`
* `docs: 更新部署文档中的环境变量说明`
* `refactor: 提取浏览器 session 管理逻辑至独立模块`

什么值得交流？
比如说一问AI就知道的东西，值得交流吗？
- 怎么提交合并请求
- 怎么处理代码仓库冲突
- 小型项目的架构
- 配置项目需要什么依赖
这些问题的答案比较固定，拿出唠嗑太浪费时间

那么，什么东西值得交流？
答案相对来说比较抽象的东西，
比如说：
- 开发某个项目的目的是什么，希望用它来实现什么。
